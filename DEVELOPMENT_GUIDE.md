# 简历智能筛选系统开发指南

本项目是一个典型的 **“数据获取 -> 数据预处理 -> 大语言模型 (LLM) 推理 -> 结果展示”** 型 AI 应用。

如果需要从零开始编写或重构这个系统，建议按照以下由底层到应用层的顺序进行开发：

## 第一步：写定义层 —— `models.py` 和 `config.py`
这是整个项目的基础和蓝图。

### 1. 先写 `models.py`（数据模型）
*   **原因**：对于任何数据处理和 AI 项目来说，**数据流是系统的主动脉**。在开始写任何逻辑之前，必须先定义好“输入”的格式（例如 `ResumeData` 类需要包含姓名、经验、原文本等字段），以及大模型的“输出”格式（例如 `MatchResult` 类包含总分、维度得分、各项评语）。只有明确了这些数据结构模型，后续编写解析器和大模型交互代码时，才会有清晰的类型提示和边界约束。

### 2. 定义 `config.py`（配置文件）
*   **原因**：项目中会涉及各种配置变量：LLM 的 API 密钥、Prompt 中用到的岗位要求（JD）、打分维度的权重占比、文件保存目录等。将这些独立成配置文件，可以避免后期代码里到处硬编码（Hardcode），大幅提升代码的灵活性和可维护性。

## 第二步：写输入与输出层 —— 简历解析与报告生成
这一步旨在让系统具备处理“脏数据”并输出“格式化数据”的能力。

### 3. 编写 `resume_parser.py`（简历解析模块）
*   **原因**：大模型通常只处理纯文本。真实的简历可能是 PDF、Word 甚至是 HTML 格式的。在将数据交给 LLM 之前，必须先解决工程问题：如何通过相关库（如 `pdfplumber`、`docx` 等）把各种杂乱的文件格式统一转换为纯文本。这个功能完全独立，且可以首先编写并进行单元测试。

### 4. 编写 `report_generator.py`（报告生成模块）
*   **原因**：根据 `models.py` 里的 `MatchResult` 最终结果，提前写好如何将数据结构导出成直观的 Excel 报表或终端高亮文本格式。这样在后续调试大模型输出时，就能直接以直观的方式查看评估结果。

## 第三步：写核心大脑 —— `llm_matcher.py`
 这是系统中最核心的功能：利用大模型分析数据。

### 5. 编写 `llm_matcher.py`（大模型匹配评估模块）
*   **原因**：此时你已经有了输入（由 `resume_parser` 提供文本）和预期输出（由 `models` 的结构定义）。这一部分的核心工作是编写 Prompt、调用 API、并处理最重要的一步——**解析 LLM 返回的结构化 JSON 并使其转化回 `MatchResult` 对象**。这是整个项目最“AI”的地方，同时也是调试花费时间最多的模块，因为需要针对大模型返回结果的不稳定性做大量的容错处理。

## 第四步：写调度与呈现层 —— `main.py` 和 `app.py`
基础模块已经准备就绪，这一步是组装业务逻辑。

### 6. 编写 `main.py`（控制台命令行入口）
*   **原因**：将之前写的散装独立模块串联起来建立数据管线（Pipeline）：`读取配置 -> 遍历目录解析简历 -> 批量送给 LLM 评分 -> 导出 Excel 报告`。用最小的代价验证系统的端到端流转是否跑得通。

### 7. 编写 `app.py`（Web 界面服务）
*   **原因**：Web UI 只是对底层业务逻辑的交互封装。在核心逻辑没有经过 `main.py` 验证之前写 API 接口容易导致前后端问题混在一起。为了降低排查难度，一定要在底层架构稳固后再搭建 Web 服务。

## 第五步（可选/扩展）：爬虫获取数据 —— `scraper.py`
这是自动化获取简历的扩展功能模块。

### 8. 编写 `scraper.py`（前程无忧爬虫）
*   **原因**：这个模块主要功能是自动化模拟获取数据的流程。即使用户自己手动放几份本地简历进入文件夹，整个系统核心依然能够运转。由于爬虫对抗（反爬、验证码、Session 保持等问题）极度消耗精力且代码量大，应该等核心的筛选评价体系（1~7 步）构建完备且跑通后，再来开发自动化获取功能。
